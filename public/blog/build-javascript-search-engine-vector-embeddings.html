How to Build a Search Engine in JavaScript (Client-Side, No Server Required)

Most JavaScript search tutorials tell you to set up Elasticsearch, spin up a Node backend, or send every query to an API. But if you're building a documentation site, a personal blog, or a tool with a few thousand searchable items, you don't need infrastructure. You need a client-side search engine that runs entirely in the browser—fast, private, and zero latency.

This is a practical guide to building semantic, vector-based search in JavaScript without backend dependencies. We'll use embeddings, cosine similarity, and a surprisingly small amount of code. By the end, you'll have a working search engine that understands "authentication error" and "login failed" are the same query, even if your content never uses those exact words.

Why Client-Side Search Beats Server Search for Small-to-Medium Datasets

Server-based search makes sense at scale. But for datasets under 10,000 documents—think documentation, product catalogs, personal archives—client-side search is faster to build, cheaper to run, and more private. No round-trips. No API keys. No logs.

The challenge is making it good. Traditional keyword search in JavaScript is trivial with String.prototype.includes(), but it's brittle. A user searching "auth problems" won't find your article titled "Authentication Troubleshooting." Fuzzy matching helps, but semantic search—using vector embeddings—solves the problem properly.

The Architecture: Embeddings, Vectors, and Cosine Similarity

Here's the simplest possible mental model:

  1. Convert each searchable document into a vector (an array of numbers representing meaning)
  2. Convert the user's query into a vector using the same method
  3. Measure similarity between query vector and document vectors
  4. Return top matches, sorted by similarity score

Vector embeddings capture semantic meaning. "Car" and "automobile" get similar vectors. "Login issue" and "authentication error" cluster close together. You're searching meaning, not just text.

Pre-compute document vectors at build time. Store them as JSON. At runtime, embed the query and calculate cosine similarity against your pre-computed vectors. On modern hardware, comparing against 5,000 vectors takes under 20ms.

Step 1: Generate Embeddings for Your Documents

You need an embedding model. The two practical options for JavaScript developers are:

Option A: Use a local model with Transformers.js
Xenova's @xenova/transformers runs ONNX models in the browser or Node. The all-MiniLM-L6-v2 model is 23MB and produces 384-dimensional vectors. Good accuracy, but the initial model load is slow.

Option B: Pre-generate embeddings at build time
Use OpenAI's API, a local model via Python, or a library like altor-vec to generate embeddings during your build process. Store them as static JSON. This is faster at runtime and keeps your bundle small.

Here's a Node script that generates embeddings for a docs site:

import { embed } from 'altor-vec';
import fs from 'fs';

const docs = [
  { id: 'auth-guide', title: 'Authentication Guide', content: '...' },
  { id: 'api-errors', title: 'Common API Errors', content: '...' },
  // ... more docs
];

const embeddings = await Promise.all(
  docs.map(async (doc) => ({
    id: doc.id,
    title: doc.title,
    vector: await embed(doc.title + ' ' + doc.content)
  }))
);

fs.writeFileSync('embeddings.json', JSON.stringify(embeddings));

Run this during build. Commit embeddings.json to your repo or upload it to your static host. Your search engine loads it at runtime.

Step 2: Implement Cosine Similarity

Cosine similarity measures how close two vectors are. Value ranges from -1 to 1. Higher is better. The formula is simple:

function cosineSimilarity(vecA, vecB) {
  let dotProduct = 0;
  let normA = 0;
  let normB = 0;
  
  for (let i = 0; i < vecA.length; i++) {
    dotProduct += vecA[i] * vecB[i];
    normA += vecA[i] * vecA[i];
    normB += vecB[i] * vecB[i];
  }
  
  return dotProduct / (Math.sqrt(normA) * Math.sqrt(normB));
}

This is the core of your search ranking. You'll call it once per document, per query. For 2,000 documents with 384-dimensional vectors, expect ~15ms on a mid-range laptop.

Step 3: Build the Search Function

Fetch your embeddings JSON. When a user types a query, embed it and rank documents by similarity:

let documentEmbeddings = [];

async function initSearch() {
  const res = await fetch('/embeddings.json');
  documentEmbeddings = await res.json();
}

async function search(query) {
  const queryVector = await embed(query);
  
  const results = documentEmbeddings
    .map(doc => ({
      ...doc,
      score: cosineSimilarity(queryVector, doc.vector)
    }))
    .sort((a, b) => b.score - a.score)
    .slice(0, 10);
  
  return results;
}

Call initSearch() on page load. Call search(userQuery) on input. Done.

Step 4: Add Hybrid Search (Keyword + Semantic)

Pure semantic search occasionally misses exact matches. If a user searches "React 18.3.1," you want exact version matches ranked higher than semantically similar content about React versions.

Combine semantic scoring with keyword scoring:

function keywordScore(query, doc) {
  const terms = query.toLowerCase().split(/\s+/);
  const text = (doc.title + ' ' + doc.content).toLowerCase();
  return terms.filter(term => text.includes(term)).length / terms.length;
}

function hybridSearch(query) {
  const queryVector = embed(query);
  
  return documentEmbeddings
    .map(doc => ({
      ...doc,
      semanticScore: cosineSimilarity(queryVector, doc.vector),
      keywordScore: keywordScore(query, doc),
      finalScore: 0.7 * semanticScore + 0.3 * keywordScore
    }))
    .sort((a, b) => b.finalScore - a.finalScore)
    .slice(0, 10);
}

The 70/30 weight works well for most use cases. Tune it based on your content.

Step 5: Optimize for Production

Three optimizations matter:

Lazy-load embeddings. Don't block page render. Fetch embeddings after DOMContentLoaded or on first focus of the search input.

Debounce queries. Don't run search on every keystroke. Wait 150-300ms after the user stops typing:

let debounceTimer;
searchInput.addEventListener('input', (e) => {
  clearTimeout(debounceTimer);
  debounceTimer = setTimeout(() => search(e.target.value), 200);
});

Use Web Workers for large datasets. If you're searching 5,000+ documents, run the similarity calculations in a worker to avoid blocking the main thread.

Real-World Constraints and When This Breaks

This approach works well up to about 10,000 documents. Beyond that, you'll feel the JSON load time (several MB) and the brute-force similarity calculation (50ms+). At that scale, you need approximate nearest neighbor search (ANN) with libraries like hnswlib-wasm, or you need a server.

Embedding quality matters more than algorithm complexity. A bad model produces random-feeling results. Test with real user queries. If "password reset" returns articles about database backups, your embeddings are undertrained for your domain.

Privacy is a feature here. Queries never leave the browser. No analytics, no logs. For internal tools, medical apps, or anything handling sensitive data, this is a significant advantage over server-based search.

Frequently Asked Questions

Can I use this for a 50,000-document site?

Not with brute-force cosine similarity. You'd need approximate nearest neighbor (ANN) indexing via hnswlib-wasm or a similar library. At that scale, a server-based solution like Typesense or Meilisearch becomes more practical.

Which embedding model should I use?

For English content, all-MiniLM-L6-v2 (via Transformers.js or sentence-transformers in Python) is the best balance of size and quality. For multilingual search, use paraphrase-multilingual-MiniLM-L12-v2. Avoid OpenAI's Ada unless you're okay with API costs at scale.

How do I handle typos?

Semantic search is somewhat typo-tolerant—"athentication" will still match "authentication" because the embedding is close—but for severe typos, add a fuzzy matching layer using Levenshtein distance or a library like Fuse.js as a fallback.

Do I need TypeScript for this?

No, but type safety helps when managing vector dimensions and similarity scores. A single dimension mismatch breaks cosine similarity silently. Types catch that at compile time.
