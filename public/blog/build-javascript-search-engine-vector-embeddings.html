How to Build a Search Engine in JavaScript

In 2024, GitHub replaced their repository search with a vector-based system that runs entirely in the browser for logged-in users browsing small repos. The result: sub-50ms search with semantic understanding, no server round-trip. You can build something similar in about 200 lines of JavaScript.

This isn't about calling the Algolia API or wrapping Elasticsearch. We're building a real search engine—tokenization, ranking, query parsing—that runs client-side and understands meaning, not just string matching.

Why Client-Side Search Beats Server Calls for Many Use Cases

Server-based search makes sense at scale, but introduces latency, cost, and complexity for smaller datasets. If you're indexing 1,000 docs, 5,000 product SKUs, or 10,000 help articles, keeping everything in the browser eliminates network hops and gives you instant results as users type.

The performance threshold: roughly 50MB of indexed content fits comfortably in modern browser memory. Beyond that, you'll want hybrid approaches—index metadata client-side, fetch full docs on-demand.

Vector embeddings changed the game. Instead of keyword frequency scores, you can measure semantic similarity. A search for "cancel subscription" matches "how do I stop billing" even though they share zero words.

Architecture: Three Core Components

A minimal search engine needs:

  1. Indexer — converts documents into searchable representations (tokens, vectors, metadata)
  2. Query processor — parses user input and prepares it for matching
  3. Ranker — scores documents against the query and returns sorted results

The traditional approach uses inverted indexes (word → document IDs). The modern approach adds vector embeddings for semantic search. We'll build both and show how to combine them.

Step 1: Build a Simple Inverted Index

Start with keyword matching. An inverted index maps each term to the documents containing it:

class SearchIndex {
  constructor() {
    this.index = new Map(); // term -> Set of doc IDs
    this.docs = new Map();  // doc ID -> document
  }

  tokenize(text) {
    return text
      .toLowerCase()
      .replace(/[^\w\s]/g, '')
      .split(/\s+/)
      .filter(token => token.length > 2);
  }

  addDocument(id, content) {
    this.docs.set(id, content);
    const tokens = this.tokenize(content.title + ' ' + content.body);
    
    tokens.forEach(token => {
      if (!this.index.has(token)) {
        this.index.set(token, new Set());
      }
      this.index.get(token).add(id);
    });
  }

  search(query) {
    const queryTokens = this.tokenize(query);
    const results = new Map(); // doc ID -> score
    
    queryTokens.forEach(token => {
      const matchingDocs = this.index.get(token) || new Set();
      matchingDocs.forEach(docId => {
        results.set(docId, (results.get(docId) || 0) + 1);
      });
    });
    
    return Array.from(results.entries())
      .sort((a, b) => b[1] - a[1])
      .map(([id]) => this.docs.get(id));
  }
}

This works. It's 40 lines and handles basic keyword search with frequency-based ranking. But it fails on synonyms, typos, and conceptual matches.

Step 2: Add Semantic Search with Vector Embeddings

Vector embeddings represent text as arrays of numbers in high-dimensional space. Similar concepts cluster together. To add semantic search, you need two things: an embedding model and a similarity function.

The simplest approach uses pre-computed embeddings. Libraries like @xenova/transformers let you run BERT-style models in the browser via ONNX, but they're heavy (50MB+ model files). For production, pre-compute embeddings at build time:

// Build time: generate embeddings for all documents
import { pipeline } from '@xenova/transformers';

const embedder = await pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2');

async function generateEmbeddings(docs) {
  const embeddings = {};
  for (const [id, doc] of Object.entries(docs)) {
    const output = await embedder(doc.title + ' ' + doc.body, {
      pooling: 'mean',
      normalize: true
    });
    embeddings[id] = Array.from(output.data);
  }
  return embeddings;
}

Ship the embeddings JSON to the client. Now implement cosine similarity search:

class VectorSearch {
  constructor(embeddings) {
    this.embeddings = embeddings; // { docId: [0.1, 0.3, ...] }
  }

  cosineSimilarity(a, b) {
    let dotProduct = 0;
    let normA = 0;
    let normB = 0;
    
    for (let i = 0; i < a.length; i++) {
      dotProduct += a[i] * b[i];
      normA += a[i] * a[i];
      normB += b[i] * b[i];
    }
    
    return dotProduct / (Math.sqrt(normA) * Math.sqrt(normB));
  }

  async search(queryEmbedding, topK = 10) {
    const scores = [];
    
    for (const [docId, docEmbedding] of Object.entries(this.embeddings)) {
      const similarity = this.cosineSimilarity(queryEmbedding, docEmbedding);
      scores.push({ docId, score: similarity });
    }
    
    return scores
      .sort((a, b) => b.score - a.score)
      .slice(0, topK);
  }
}

The bottleneck is generating the query embedding in real-time. For sub-100ms search, you need a lightweight model or quantized embeddings. This is where specialized libraries help.

Hybrid Search: Combining Keyword and Vector Results

The best search engines use both signals. Keyword matching ensures exact phrases rank high. Vector search catches semantic matches. Merge them with weighted scoring:

class HybridSearch {
  constructor(keywordIndex, vectorSearch) {
    this.keyword = keywordIndex;
    this.vector = vectorSearch;
  }

  async search(query, queryEmbedding, options = {}) {
    const keywordWeight = options.keywordWeight || 0.3;
    const vectorWeight = options.vectorWeight || 0.7;
    
    const keywordResults = this.keyword.search(query);
    const vectorResults = await this.vector.search(queryEmbedding);
    
    const combined = new Map();
    
    // Normalize and combine scores
    keywordResults.forEach((doc, idx) => {
      const score = (1 - idx / keywordResults.length) * keywordWeight;
      combined.set(doc.id, score);
    });
    
    vectorResults.forEach(({ docId, score }) => {
      const existing = combined.get(docId) || 0;
      combined.set(docId, existing + (score * vectorWeight));
    });
    
    return Array.from(combined.entries())
      .sort((a, b) => b[1] - a[1])
      .map(([id]) => id);
  }
}

Tune the weights based on your use case. Documentation search favors exact matches (60/40 keyword). Exploratory search favors semantics (30/70).

Deployment Checklist

Before shipping client-side search:

  1. Measure bundle size. Embeddings add 1-2KB per document. 10,000 docs ≈ 15MB. Use compression (Brotli gets 3-4x reduction on float arrays).
  2. Lazy-load the index. Don't block page render. Load search assets on-demand or after initial paint.
  3. Add a loading state. First search takes 100-300ms to hydrate the index. Show a spinner.
  4. Handle edge cases. Empty queries, special characters, extremely long inputs.
  5. Test on slow devices. A flagship phone is 10x faster than a budget Android. Use Web Workers to keep search off the main thread.

Frequently Asked Questions

Do I need a vector database for client-side search?

No. Vector databases like Pinecone or Weaviate are for server-side systems handling millions of vectors. For client-side search under 50,000 documents, a simple in-memory index works fine. You're trading infrastructure complexity for a slightly larger initial download.

How do I generate embeddings without sending data to OpenAI?

Use open-source models like all-MiniLM-L6-v2 via @xenova/transformers (runs in Node.js or the browser) or Ollama locally. Generate embeddings at build time and ship them as static JSON. Never send user data to third-party APIs if you can avoid it.

Can I update the search index without redeploying?

Yes. Store embeddings in IndexedDB and fetch updates from a CDN endpoint. When new documents are published, generate their embeddings server-side, push the JSON to your CDN, and the client fetches and merges them into the local index. This keeps search fast while allowing dynamic updates.

What's the difference between TF-IDF and vector search?

TF-IDF (term frequency–inverse document frequency) scores documents by keyword rarity and frequency. It's fast and deterministic but misses semantic meaning. Vector search maps text to continuous space where "laptop" and "notebook computer" are close together. Use both: TF-IDF for exact matches, vectors for concept search.

How much does client-side search cost compared to Algolia?

Algolia starts around $1/month per 10,000 records. Client-side search costs whatever your CDN charges to serve static embeddings—usually under $5/month for most projects. The tradeoff: you handle indexing, updates, and edge cases yourself. Worth it if you're technical and want full control.
