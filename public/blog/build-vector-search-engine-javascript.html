How to Build a Search Engine in JavaScript

Most tutorials on building a search engine in JavaScript end with a basic string-matching function that breaks the moment your dataset exceeds 100 items or users type something slightly off from your indexed terms. You get a working demo, sure, but not something you'd ship to production. The real challenge isn't making search work—it's making it feel intelligent without sending every keystroke to a remote API.

This guide walks through building a client-side vector search engine that runs entirely in the browser. No backend calls. No latency. Just semantic understanding of what users actually mean, powered by embeddings and cosine similarity. We'll use JavaScript, the Web Worker API for performance, and a lightweight vector library to handle the math.

Why Vector Search Beats String Matching

Traditional search relies on exact or fuzzy text matches. Search for "JS frameworks" and you'll miss documents that say "JavaScript libraries" or "React ecosystems." Vector search converts text into numerical representations—embeddings—that capture semantic meaning. Documents about similar concepts cluster together in high-dimensional space, even if they use completely different words.

The trade-off: you need to generate embeddings for your content upfront. But once you have them, search becomes a simple distance calculation. No regex parsing, no inverted indices, no fragile synonym dictionaries.

This approach makes sense when you have 100 to 50,000 documents, you control the content pipeline, and you want sub-50ms query response times. Think documentation sites, internal wikis, product catalogs, or any app where users expect instant filtering.

Architecture Overview

Our search engine has three pieces:

Embedding generation. At build time, convert each document into a vector using a model like OpenAI's text-embedding-3-small or Cohere's embed-light. Store these vectors in a JSON file bundled with your app.

Query encoding. When a user types a search term, encode it into a vector using the same model. This happens client-side if you're using a WASM-based model, or via a single API call if you're using a hosted service.

Similarity ranking. Calculate cosine similarity between the query vector and every document vector. Sort by score. Return top N results.

The entire process runs in a Web Worker so the UI stays responsive during large searches.

Step 1: Generate Embeddings for Your Content

Start with a Node script that reads your documents and calls an embedding API. Here's a minimal version using OpenAI:

import fs from 'fs';
import OpenAI from 'openai';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

const documents = [
  { id: 1, title: 'React Server Components', text: 'RSCs let you fetch data...' },
  { id: 2, title: 'Vite vs Webpack', text: 'Build tools compared...' },
  // ... more docs
];

async function generateEmbeddings() {
  const embeddings = [];
  
  for (const doc of documents) {
    const response = await openai.embeddings.create({
      model: 'text-embedding-3-small',
      input: `${doc.title} ${doc.text}`,
    });
    
    embeddings.push({
      id: doc.id,
      vector: response.data[0].embedding,
      metadata: { title: doc.title },
    });
  }
  
  fs.writeFileSync('embeddings.json', JSON.stringify(embeddings));
}

generateEmbeddings();

Run this during your build step. The output is a single JSON file with document IDs, vectors (arrays of 1536 floats for text-embedding-3-small), and any metadata you want to display in results.

Cost: roughly $0.02 per 1,000 documents with OpenAI's smallest model. Re-run only when content changes.

Step 2: Load Embeddings in the Browser

Import the embeddings file and initialize your search index. If you're bundling with Vite or Next.js, you can import JSON directly:

import embeddingsData from './embeddings.json';

class VectorSearch {
  constructor(embeddings) {
    this.embeddings = embeddings;
  }
  
  cosineSimilarity(vecA, vecB) {
    let dotProduct = 0;
    let normA = 0;
    let normB = 0;
    
    for (let i = 0; i < vecA.length; i++) {
      dotProduct += vecA[i] * vecB[i];
      normA += vecA[i] * vecA[i];
      normB += vecB[i] * vecB[i];
    }
    
    return dotProduct / (Math.sqrt(normA) * Math.sqrt(normB));
  }
  
  search(queryVector, topK = 10) {
    const results = this.embeddings
      .map(doc => ({
        ...doc,
        score: this.cosineSimilarity(queryVector, doc.vector),
      }))
      .sort((a, b) => b.score - a.score)
      .slice(0, topK);
    
    return results;
  }
}

const searchEngine = new VectorSearch(embeddingsData);

This class handles the core ranking logic. The cosineSimilarity function is the entire "algorithm"—dot product divided by magnitude product. Simple enough that you could write it in a job interview.

Step 3: Encode User Queries

When the user types a search, you need to convert their query string into a vector. You have two options:

Option A: API call. Send the query to your embedding provider. Adds 100-300ms latency but requires zero client-side compute.

async function encodeQuery(query) {
  const response = await fetch('https://api.openai.com/v1/embeddings', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${apiKey}`,
    },
    body: JSON.stringify({
      model: 'text-embedding-3-small',
      input: query,
    }),
  });
  
  const data = await response.json();
  return data.data[0].embedding;
}

Option B: Local model. Use a WASM-compiled model like Transformers.js or ONNX Runtime Web. Runs entirely in the browser but requires loading a 20-40MB model file on first use.

import { pipeline } from '@xenova/transformers';

let embedder;

async function initEmbedder() {
  embedder = await pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2');
}

async function encodeQuery(query) {
  const output = await embedder(query, { pooling: 'mean', normalize: true });
  return Array.from(output.data);
}

For most use cases, the API approach is faster unless users are making dozens of searches per session. The hybrid option: cache the model in IndexedDB after first load.

Step 4: Wire It Up with a Web Worker

Searching through 10,000+ vectors can block the main thread. Offload the work to a Web Worker:

// search-worker.js
import embeddingsData from './embeddings.json';

const searchEngine = new VectorSearch(embeddingsData);

self.addEventListener('message', (e) => {
  const { queryVector } = e.data;
  const results = searchEngine.search(queryVector, 10);
  self.postMessage(results);
});

// main.js
const worker = new Worker(new URL('./search-worker.js', import.meta.url), { type: 'module' });

async function performSearch(query) {
  const queryVector = await encodeQuery(query);
  
  return new Promise((resolve) => {
    worker.onmessage = (e) => resolve(e.data);
    worker.postMessage({ queryVector });
  });
}

Now your search runs in parallel with UI rendering. Users can type freely without jank.

Optimizations Worth Considering

Once the basic engine works, there are three upgrades that make a noticeable difference:

Quantization. Reduce vector precision from 32-bit floats to 8-bit integers. Cuts storage by 75% and speeds up similarity calculations. Accuracy drop is negligible for most use cases. The altor-vec library handles this automatically with quantize: true in the config.

Hybrid scoring. Combine vector similarity with traditional keyword matching for best results. Weight them 70/30. This catches exact product codes or acronyms that embeddings sometimes blur.

Prefiltering. If you have metadata like categories or dates, filter the document set before calculating similarity. Searching 1,000 docs is faster than 10,000, and users often want scoped results anyway.

When This Approach Fails

Client-side vector search stops making sense above 100,000 documents. The embedding file becomes too large to bundle (20-30MB+), and even with Web Workers, similarity calculations start taking 500ms+. At that scale, reach for a proper vector database like Pinecone, Weaviate, or Qdrant running server-side with approximate nearest neighbor algorithms.

It also struggles with multilingual content unless you use a multilingual embedding model. And if your content updates frequently—several times per day—you'll need a build pipeline that regenerates embeddings fast.

Alternative: Skip the Complexity

If you want vector search without writing the cosine similarity loop yourself, altor-vec packages this entire pattern into a 4KB library. It handles quantization, hybrid scoring, and Web Worker orchestration:

import { createSearchEngine } from 'altor-vec';

const engine = await createSearchEngine({
  embeddings: './embeddings.json',
  quantize: true,
  hybrid: { keywords: 0.3 },
});

const results = await engine.search('react state management');

The API is deliberately minimal. You provide embeddings, it provides results. No configuration sprawl, no framework lock-in.

Frequently Asked Questions

Can I use this for fuzzy text search without embeddings?

Technically yes—use TF-IDF vectors instead of semantic embeddings—but at that point libraries like Fuse.js or FlexSearch are simpler. Vector search shines when you need semantic understanding, not just typo tolerance.

What's the minimum number of documents where this makes sense?

Around 50. Below that, a basic filter function is faster to implement and good enough. The overhead of generating embeddings and setting up workers isn't worth it for tiny datasets.

How do I handle search result relevance tuning?

Log queries and clicks. If certain documents rank high but never get clicked, either their metadata is misleading or the embedding model isn't capturing the right semantics. Sometimes manually boosting specific docs by 0.1-0.2 in the similarity score fixes edge cases faster than retraining models.

Does this work offline?

Completely, if you use a local embedding model and bundle the vectors with your app. The entire search stack runs without network access. Perfect for Electron apps or PWAs.

What about privacy?

User queries never leave the browser unless you choose the API-based encoding route. If privacy matters, use Transformers.js for local encoding and keep everything client-side.

Need vector search without the setup overhead? npm install altor-vec gives you a production-ready implementation in one line. Check the repo at github.com/Altor-lab/altor-vec for API docs and advanced configuration examples.
