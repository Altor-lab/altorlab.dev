Client-Side Vector Search Guide: Build Semantic Search Without a Backend

In March 2024, a developer at a YC-backed startup shipped a 12MB JavaScript bundle to production. Half of it was a vector search engine running entirely in the browser. The feature? A semantic search bar that let users find documents by meaning, not keywords—no server required, no API keys, zero latency. It worked offline. It cost nothing to run at scale.

Client-side vector search is no longer a novelty. With WebAssembly, ONNX Runtime, and libraries like Transformers.js, you can embed a full semantic search pipeline in under 50 lines of code. The challenge isn't whether you can do it—it's knowing when you should, and how to avoid the bundle size trap.

Why Run Vector Search in the Browser

The standard architecture for semantic search involves a backend service: documents get chunked, embedded via OpenAI or Cohere, stored in Pinecone or Weaviate, and queried server-side. This works, but it introduces costs, latency, and infrastructure complexity.

Client-side search flips the model. You pre-compute embeddings at build time, ship them as static JSON or binary files, and let the browser handle cosine similarity at runtime. The tradeoffs are sharp:

Wins: Zero API costs. Sub-100ms query latency. Works offline. No rate limits. Deploy to a CDN and forget about it.

Losses: Bundle size becomes your bottleneck. No live updates without redeploying. Limited to datasets under ~10,000 items unless you get clever with lazy loading.

The sweet spot: documentation sites, internal tools, personal knowledge bases, browser extensions, and any product where you control the corpus and users value speed over freshness.

The Minimal Implementation

Here's the simplest version using Transformers.js and a pre-embedded dataset. This example assumes you've already generated embeddings offline—we'll cover that next.

import { pipeline } from '@xenova/transformers';

// Load a 384-dimensional embedding model (22MB gzipped)
const extractor = await pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2');

// Your pre-computed document embeddings
const documents = [
  { id: 1, text: 'React hooks let you use state in function components', embedding: [...] },
  { id: 2, text: 'Vue Composition API provides reactive primitives', embedding: [...] },
  // ... 500 more
];

async function search(query) {
  const queryEmbedding = await extractor(query, { pooling: 'mean', normalize: true });
  const queryVector = Array.from(queryEmbedding.data);
  
  const results = documents.map(doc => ({
    ...doc,
    score: cosineSimilarity(queryVector, doc.embedding)
  })).sort((a, b) => b.score - a.score);
  
  return results.slice(0, 10);
}

function cosineSimilarity(a, b) {
  const dot = a.reduce((sum, val, i) => sum + val * b[i], 0);
  const magA = Math.sqrt(a.reduce((sum, val) => sum + val * val, 0));
  const magB = Math.sqrt(b.reduce((sum, val) => sum + val * val, 0));
  return dot / (magA * magB);
}

This works. Type a query, get ranked results in ~80ms. But there's a problem: that 22MB model gets downloaded on every page load. For a documentation site, that's acceptable. For a marketing page, it's a dealbreaker.

Optimizing Bundle Size Without Sacrificing Quality

The model is your biggest cost. Transformers.js ships ONNX models that are already quantized, but you can shrink further:

1. Use a smaller model. all-MiniLM-L6-v2 (22MB) is the default, but gte-small (14MB) performs nearly as well on English text. For single-language use cases, try distilled models under 10MB.

2. Lazy-load the model. Don't import it on page load. Trigger download only when the user focuses the search input:

let extractorPromise = null;

searchInput.addEventListener('focus', () => {
  if (!extractorPromise) {
    extractorPromise = pipeline('feature-extraction', 'Xenova/gte-small');
  }
}, { once: true });

3. Pre-compute everything you can. If your corpus is static (docs, blog posts, product catalog), generate embeddings at build time with a Node.js script. Ship only the vectors, not the raw text:

// build-embeddings.mjs
import { pipeline } from '@xenova/transformers';
import fs from 'fs';

const extractor = await pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2');
const docs = JSON.parse(fs.readFileSync('docs.json'));

const embedded = await Promise.all(docs.map(async (doc) => {
  const output = await extractor(doc.text, { pooling: 'mean', normalize: true });
  return { id: doc.id, text: doc.text, embedding: Array.from(output.data) };
}));

fs.writeFileSync('embeddings.json', JSON.stringify(embedded));

Run this once. Deploy embeddings.json (typically 200-500KB for 1,000 documents). The client only needs to load the model when a user actually searches.

When to Use a Lightweight Alternative

If 10MB+ is still too heavy, consider altor-vec, a purpose-built library for client-side vector search. It ships with a 2.4MB model and a compact HNSW index that handles 50,000 documents without breaking a sweat.

npm install altor-vec

The API is leaner:

import { VectorSearch } from 'altor-vec';

const search = new VectorSearch({
  model: 'mini',
  index: await fetch('/embeddings.bin').then(r => r.arrayBuffer())
});

const results = await search.query('how to use react hooks');
// Returns ranked results in <50ms

This approach trades some embedding quality for drastically smaller payloads. For most developer-facing search (documentation, changelogs, code snippets), the difference in recall is negligible.

Handling Updates and Cache Invalidation

Static embeddings mean your search index doesn't update until you redeploy. For documentation sites that publish weekly, this is fine. For user-generated content, it's a nonstarter.

A hybrid approach: serve recent content (last 30 days) from a real-time API, older content from the cached client-side index. Use a simple score merge:

const [cachedResults, liveResults] = await Promise.all([
  clientSearch.query(query),
  fetch(`/api/search?q=${query}`).then(r => r.json())
]);

const merged = [...liveResults, ...cachedResults]
  .sort((a, b) => b.score - a.score)
  .slice(0, 20);

Cache the static embeddings aggressively (one year TTL). Version the filename (embeddings.v2.bin) and update the reference when you rebuild.

Performance Benchmarks You Should Know

On a 2021 M1 MacBook Pro, searching 5,000 documents with cosine similarity takes ~12ms. The bottleneck is almost always model inference (~60-80ms for Transformers.js, ~30ms for altor-vec). Lazy-loading cuts perceived latency from 2 seconds to ~100ms.

Mobile is slower. On an iPhone 13, expect ~120ms for inference. Android mid-range devices can hit 200ms+. Always test on real hardware, not DevTools throttling.

When Client-Side Search Is the Wrong Choice

Don't use client-side vector search if:

Your corpus exceeds 20,000 items and updates frequently. You'll blow past reasonable bundle sizes and cache management becomes hell.

You need sub-50ms query times on every device. Server-side search with a warm cache will always be faster at scale.

You're building for regions with slow connections. A 15MB initial download is a bad trade when your users are on 3G.

Your product requires audit logs or usage analytics per query. Client-side means you lose visibility unless you build telemetry, which adds complexity.

FAQ

Can I use client-side vector search with server-side rendering?

Yes, but the model only runs in the browser. You can pre-render the page with SSR/SSG and lazy-load the search feature on the client. The embeddings JSON can be fetched as a static asset.

How do I avoid shipping duplicate embeddings if my pages share content?

Deduplicate at build time. Hash each chunk of text, store embeddings in a lookup table, and reference them by ID in your documents array. This can cut payload size by 40-60% for sites with repeated boilerplate.

Does this work with non-English text?

Most multilingual models (like paraphrase-multilingual-MiniLM) are 120MB+, which breaks the client-side budget. If you only need one non-English language, use a distilled model for that language specifically—often under 20MB.

What's the best way to debug relevance issues?

Log the cosine scores. Anything below 0.3 is usually noise. If relevant results score under 0.5, your embeddings might be stale or your query phrasing is too abstract. Test with exact phrases from your docs first, then expand to paraphrases.

Client-side vector search isn't the default choice, but for the right use case—static corpus, speed-sensitive users, cost-conscious teams—it's absurdly effective. Start with a lazy-loaded model, pre-compute your embeddings, and ship the whole thing to a CDN.
