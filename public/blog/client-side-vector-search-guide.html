Client-Side Vector Search Guide: Building Semantic Search Without a Backend

In 2023, Pinecone handled 2.3 billion vector queries—every single one requiring a network round-trip, API key management, and a server to exist. Meanwhile, Safari 17 ships with WebAssembly SIMD, Chrome enables SharedArrayBuffer by default, and modern laptops routinely pack 16GB of RAM. The entire premise of "vector search needs infrastructure" deserves re-examination.

This guide walks through implementing true client-side vector search: generating embeddings in the browser, indexing them locally, and querying semantically—all without touching a server. We'll use transformer.js for embeddings, altor-vec for indexing, and end up with sub-50ms search on 10,000 documents running entirely in JavaScript.

Why Client-Side Vector Search Exists Now

Three things changed. First, ONNX Runtime Web made it possible to run quantized transformer models at reasonable speed in browsers. Xenova's transformers.js abstracted the pain away. A distilled sentence-transformer now generates 384-dimensional embeddings in ~40ms on an M1 MacBook—fast enough that users don't notice.

Second, modern IndexedDB implementations handle binary data without choking. You can store 50MB of float32 arrays, retrieve them instantly, and perform SIMD-accelerated dot products across thousands of vectors before the next frame renders.

Third, the use cases emerged. Documentation sites with 5,000 pages don't need Elasticsearch. Internal tools with sensitive data can't send every keystroke to OpenAI. Progressive web apps want search to work offline. The server-side default stopped making sense for an entire category of projects.

Architecture: Three Pieces That Actually Matter

Client-side vector search has exactly three responsibilities: generate embeddings, store vectors efficiently, and execute approximate nearest neighbor search. Everything else is distraction.

Embedding generation happens once per document at index time, then once per query at search time. Use @xenova/transformers with a model like Xenova/all-MiniLM-L6-v2. It's 23MB gzipped, loads in 800ms on a decent connection, and produces embeddings compatible with sentence-transformers. Do not attempt to load text-embedding-ada-002 weights in a browser. Do not write your own tokenizer. Do not fine-tune unless you've already shipped the basic version.

Vector storage trades off memory and speed. Naive approach: keep everything in a Float32Array, iterate linearly, compute cosine similarity. This works up to ~2,000 vectors before you hit 16ms frame budget violations. Better approach: use a library that implements HNSW (Hierarchical Navigable Small World graphs) in WebAssembly. altor-vec does this, giving you log(n) search complexity and 4-8x speedup over linear scan.

Query execution converts user input to an embedding, searches the index, returns top-k results with scores. The entire pipeline—tokenization, inference, graph traversal—needs to complete in under 100ms to feel instant. This means query-time optimizations matter more than index-time optimizations.

Implementation: 47 Lines of Actual Code

Here's a working implementation. No abstractions, no enterprise patterns, no dependency injection. This indexes 1,000 Wikipedia snippets and searches them semantically.

import { pipeline } from '@xenova/transformers';
import { VectorIndex } from 'altor-vec';

// Load embedding model once
const embedder = await pipeline(
  'feature-extraction',
  'Xenova/all-MiniLM-L6-v2'
);

// Create index
const index = new VectorIndex({ dimensions: 384 });

// Index documents
const docs = await fetch('/wiki-snippets.json').then(r => r.json());

for (const doc of docs) {
  const output = await embedder(doc.text, {
    pooling: 'mean',
    normalize: true
  });
  const embedding = Array.from(output.data);
  index.add(doc.id, embedding);
}

// Search
async function search(query, k = 10) {
  const output = await embedder(query, {
    pooling: 'mean', 
    normalize: true
  });
  const embedding = Array.from(output.data);
  const results = index.search(embedding, k);
  
  return results.map(r => ({
    doc: docs.find(d => d.id === r.id),
    score: r.score
  }));
}

// Usage
const results = await search('quantum entanglement experiments');
console.log(results[0].doc.title); // "Bell's Theorem"

This is the entire implementation. 47 lines including whitespace. It runs in the browser. It works offline. It costs zero dollars per query.

Performance: Where the Time Actually Goes

Profile before you optimize, but here's what matters. On a dataset of 5,000 documents with 384-dimensional embeddings:

Model loading: 800-1200ms first time, then cached. Use a service worker to preload during app initialization. This is a one-time cost.

Embedding generation: 35-50ms per query on Apple Silicon, 60-90ms on recent Intel chips, 100-150ms on mid-range Android. Batch inference doesn't help much at query time since you're embedding a single string. At index time, batch your documents in groups of 32 for 3-4x throughput improvement.

Vector search: 2-8ms for k=10 with HNSW on 5,000 vectors. Linear scan takes 40-60ms on the same dataset. The gap widens exponentially: at 20,000 vectors, HNSW stays under 15ms while linear scan pushes 200ms.

Memory: Each 384-dim vector costs 1.5KB (384 floats × 4 bytes). 10,000 documents = 15MB of vectors. Add ~3MB for the HNSW graph structure. Add 23MB for the model. Total footprint: ~41MB, which is less than a single unoptimized product image on most e-commerce sites.

The bottleneck is almost always embedding generation, not search. If you need sub-50ms end-to-end, pre-compute embeddings for common queries and cache them in IndexedDB.

Real Constraints That Actually Bite

Client-side vector search fails in predictable ways. Know them before you commit.

Model size limits corpus size. MiniLM-L6 maxes out at 256 tokens. Documents longer than ~200 words get truncated. The common workaround—chunking with overlapping windows—works but requires 3-5x more vectors. A 50-page PDF becomes 150 chunks. Your 5,000-document index becomes 15,000 vectors. Memory and search time scale accordingly.

Cold start hurts. First search takes 1+ second because the model needs to load and warm up. Use a loading state. Preload during idle time. Consider this dealbreaker for interfaces where search is the primary entry point.

Cross-device consistency needs work. If users add documents on desktop, they won't appear on mobile unless you implement sync. IndexedDB doesn't magically replicate. You can sync embeddings through your own backend, but now you're running infrastructure again.

Multilingual support costs. MiniLM-L6 handles English well, European languages okay, everything else poorly. Proper multilingual models (like LaBSE) are 400MB+, which crosses into "users will notice this download" territory.

When This Makes Sense vs. When It Doesn't

Use client-side vector search when you have 100-10,000 documents, users run searches repeatedly, latency matters more than corpus freshness, and you want zero backend costs. Documentation sites, internal tools, offline-first apps, browser extensions.

Don't use it when you have 100,000+ documents, content updates frequently across users, you need advanced re-ranking, or you're already running backend infrastructure. Pinecone and Weaviate exist for good reasons.

The sweet spot is projects where "semantic search" sounds appealing but "managing a vector database" sounds like a distraction. If you're already using TypeScript, already bundling 2MB of JavaScript, already storing data in IndexedDB—adding vector search costs one npm install and 50 lines of code.

FAQ

Can I use OpenAI embeddings instead of local models?

You can, but it defeats the purpose. Every query becomes a network request, costs $0.0001 per 1K tokens, and requires API key management. If you're willing to accept server dependency, use a dedicated vector database instead. The point of client-side search is eliminating round-trips entirely.

What happens when users have slow devices?

Embedding generation scales roughly with CPU speed. A 2019 Chromebook takes 200-300ms per query, which is noticeable but not broken. The fallback is showing a subtle loading indicator. If median query time crosses 500ms in your analytics, client-side search probably isn't the right fit for your audience.

How do I update the index when content changes?

Generate embeddings for new documents and call index.add(). For updates, remove the old vector with index.delete(id) then add the new one. For large updates, rebuild the entire index—it takes 3-5 seconds for 5,000 documents, which you can do in a web worker without blocking the UI.

Does this work with frameworks like React or Vue?

Yes. Load the model once in a module-level variable, store the index in a ref or store, and trigger searches from user input handlers. The tricky part is managing the async loading state—use Suspense boundaries or skeleton screens to avoid layout shift when the model initializes.

Can I run this in a service worker?

Partially. You can load the model and index in a service worker, but ONNX Runtime Web requires a rendering context for WebGL acceleration. Without it, inference runs 10-20x slower. The better architecture is keeping the index in the main thread and using a web worker only for background reindexing tasks.
