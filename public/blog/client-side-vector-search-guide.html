Client-Side Vector Search Guide: Build Semantic Search in 200 Lines of JavaScript

Most vector search tutorials end with "now deploy a Pinecone cluster" or "spin up a Weaviate instance." But 80% of production search features don't need a database. They need 10,000 embedded documents, a cosine similarity loop, and 14ms query time—all running in the browser.

Client-side vector search means your search index lives in IndexedDB, your embedding model runs in WebAssembly, and your nearest-neighbor algorithm executes on the main thread without blocking the UI. No API keys. No rate limits. No server round-trip adding 200ms to every keystroke.

This guide walks through the actual implementation: how to embed documents with transformers.js, store vectors efficiently, and query them fast enough for autocomplete. Why Client-Side Search Outperforms Server Search for Small Corpora

The math is simple. A round-trip to your edge function takes 80-150ms under good conditions. A brute-force cosine similarity search through 10,000 384-dimensional vectors takes 8ms in modern JavaScript engines.

Vercel's Edge Runtime adds 60-120ms. Cloudflare Workers add 40-90ms. Supabase pgvector adds 30-200ms depending on your plan and region. Meanwhile, IndexedDB retrieval is 2-5ms and in-memory vector comparison is nearly free at that scale.

The crossover point is around 100,000 documents. Below that, client-side wins on latency. Above that, you need HNSW indexes and GPU acceleration that only server-side libraries provide. But most product search, docs search, and internal tool search never hit 100k documents.

The Three-Layer Stack

Every client-side vector search implementation has three parts: an embedding model that converts text to vectors, a storage layer that persists those vectors between sessions, and a query engine that finds nearest neighbors fast enough for interactive use.

Layer 1: Embeddings with transformers.js. Xenova's transformers.js library wraps HuggingFace models in a WebWorker-friendly API. The all-MiniLM-L6-v2 model produces 384-dimensional embeddings at 15ms per sentence on a 2020 MacBook Pro. That's the baseline. Newer quantized models like gte-small hit 8ms but sacrifice 3-5 points of retrieval accuracy.

Install it: npm install @xenova/transformers. The first load downloads a 23MB ONNX model to browser cache. Subsequent page loads are instant.

Layer 2: IndexedDB for vector persistence. LocalStorage caps at 5-10MB. IndexedDB gives you 50MB minimum, often 500MB-1GB depending on browser and user storage quotas. Store vectors as Float32Arrays wrapped in structured objects. A 10,000-document corpus with metadata uses roughly 18MB.

Layer 3: Brute-force cosine similarity with TypedArray optimizations. You don't need HNSW or LSH at 10k documents. A tight loop over Float32Arrays with manual dot product calculation runs fast enough. The trick is doing it without blocking the main thread.

Implementation: A Working 200-Line Prototype

Here's the minimal viable version. This code embeds a document array, stores vectors in IndexedDB, and queries them in under 15ms for a 5,000-doc corpus.

import { pipeline } from '@xenova/transformers';

class ClientVectorSearch {
  constructor() {
    this.embedder = null;
    this.db = null;
    this.vectors = [];
  }

  async init() {
    this.embedder = await pipeline('feature-extraction', 'Xenova/all-MiniLM-L6-v2');
    this.db = await this.openDB();
    this.vectors = await this.loadVectors();
  }

  async embed(text) {
    const output = await this.embedder(text, { pooling: 'mean', normalize: true });
    return Array.from(output.data);
  }

  async indexDocuments(docs) {
    const tx = this.db.transaction('vectors', 'readwrite');
    for (const doc of docs) {
      const vector = await this.embed(doc.text);
      await tx.store.put({ id: doc.id, vector, metadata: doc });
    }
    await tx.done;
    this.vectors = await this.loadVectors();
  }

  cosineSimilarity(a, b) {
    let dot = 0;
    for (let i = 0; i < a.length; i++) dot += a[i] * b[i];
    return dot; // Assumes normalized vectors
  }

  async search(query, topK = 10) {
    const queryVec = await this.embed(query);
    const results = this.vectors.map(item => ({
      ...item.metadata,
      score: this.cosineSimilarity(queryVec, item.vector)
    }));
    return results.sort((a, b) => b.score - a.score).slice(0, topK);
  }

  async openDB() {
    return new Promise((resolve, reject) => {
      const req = indexedDB.open('VectorSearchDB', 1);
      req.onupgradeneeded = () => req.result.createObjectStore('vectors', { keyPath: 'id' });
      req.onsuccess = () => resolve(req.result);
      req.onerror = () => reject(req.error);
    });
  }

  async loadVectors() {
    const tx = this.db.transaction('vectors', 'readonly');
    return await tx.store.getAll();
  }
}

This is production-adjacent code. It needs error handling, a Web Worker wrapper, and incremental indexing for large corpora, but the core loop is here. Notice the normalized vectors: we skip magnitude calculation in cosine similarity because transformers.js outputs unit vectors when you pass normalize: true.

Performance Profiling: Where 10k Documents Becomes 100k

At 10,000 documents, search takes 8-12ms. At 50,000, it's 35-50ms. At 100,000, you're at 90-140ms and users feel the lag. The bottleneck is the linear scan through every vector on every keystroke.

Three fixes scale you past 100k without reaching for a server:

1. Move search to a Web Worker. The main thread stays responsive. The worker handles the O(n) loop. Message-passing overhead is 1-2ms, negligible compared to the 40ms you save by not blocking the UI.

2. Quantize vectors from Float32 to Int8. An 384-float vector is 1.5KB. An Int8 version is 384 bytes. You fit 4x more vectors in memory and IndexedDB. Retrieval accuracy drops 2-3 percentage points but query speed doubles because cache lines fit more data.

3. Add a coarse filter. Embed cluster centroids (e.g., 100 clusters for 100k docs using k-means). At query time, find the top 3 nearest clusters, then search only documents in those clusters. This cuts the search space by 97% with minimal accuracy loss.

With all three optimizations, you can realistically handle 200,000 documents client-side. Beyond that, you need HNSW, and HNSW in JavaScript is slower than HNSW in Rust over a network call.

The Libraries You Actually Need

Transformers.js is the obvious choice for embeddings. It's actively maintained, supports 50+ models, and runs in any modern browser. The alternative is TensorFlow.js, which has a steeper learning curve and worse WebAssembly performance for NLP tasks.

For vector storage, raw IndexedDB is fine under 20k documents. Above that, consider Dexie.js for a cleaner API. It adds 20KB gzipped but eliminates 80% of IndexedDB's callback hell.

For the search layer, write it yourself. Vector search libraries like hnswlib-wasm or usearch add 200KB+ for features you don't need at small scale. A 50-line cosine similarity loop outperforms them under 50k documents.

FAQ

How do I handle embedding updates when documents change?

Maintain a version hash (e.g., MD5 of document content) alongside each vector in IndexedDB. On app load, compare hashes. Re-embed only changed documents. For 10k documents, a full re-index takes 2-3 minutes. Incremental updates take 50-200ms depending on how many docs changed.

Can I run this on mobile browsers?

Yes, but Safari's IndexedDB quota is stricter (50MB default vs. Chrome's 500MB+). Transformers.js works on iOS 16+ but runs 2-3x slower than desktop. Keep your corpus under 20k documents on mobile or aggressively quantize vectors to Int8.

What about privacy and offline-first apps?

Client-side vector search is ideal for privacy-sensitive apps. Embeddings never leave the device. Search queries aren't logged. The entire index lives in browser storage. This is why Obsidian and other local-first tools are moving toward client-side semantic search.

How accurate is this compared to server-side pgvector or Pinecone?

For exact nearest-neighbor search under 50k documents, accuracy is identical—both use brute-force cosine similarity. Approximate methods like HNSW trade 1-5% recall for 100x speed, but you don't need them client-side until you hit 100k+ documents.
