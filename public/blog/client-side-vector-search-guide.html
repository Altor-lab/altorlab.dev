Client-Side Vector Search Guide: Building Semantic Search Without a Backend

Most vector search tutorials assume you have a server. They walk you through setting up Pinecone, Weaviate, or Qdrant, spinning up an API, managing API keys, and dealing with cold starts. But what if your search index fits in 50MB and your users already downloaded it when they loaded your documentation site?

Client-side vector search runs entirely in the browser. No server. No API calls. No latency spikes when your backend decides to take a nap. For documentation sites, internal tools, offline-first apps, and anything where your corpus is small enough to ship as a static asset, it's faster and dramatically simpler than reaching for a vector database.

Why Client-Side Vector Search Works

The typical objections are predictable: "Embeddings are too large." "JavaScript is too slow." "Users won't download megabytes of vectors."

Here's what actually happens. A 1,000-document corpus with 384-dimensional embeddings (the size you get from models like MiniLM) takes roughly 1.5MB. Gzipped, that drops to around 400KB. Your average SaaS marketing site ships more JavaScript than that in analytics trackers alone.

Modern browsers handle typed arrays and SIMD operations well enough that cosine similarity across a few thousand vectors completes in under 10ms on a mid-range laptop. That's faster than a round trip to a server in the same city.

The real constraint is corpus size. If you're indexing 100,000 product descriptions, you need a server. If you're indexing 2,000 help articles, a few hundred API docs, or a static site's content, client-side search is not just viable—it's the better architecture.

How It Works: Embeddings, Indexes, and Similarity

Vector search reduces text to fixed-length arrays of numbers (embeddings) that capture semantic meaning. "refund policy" and "how do I get my money back" produce similar vectors even though they share no words. When a user searches, you convert their query to a vector and find the closest matches by measuring distance.

The process has three stages. First, generate embeddings for your documents offline using a model like sentence-transformers/all-MiniLM-L6-v2. You do this once during your build step, not at runtime. Second, serialize those embeddings into a compact format and ship them as a static file. Third, at runtime, load the embeddings, convert the user's query to a vector (using a WASM-based model or a lightweight transformer), and compute cosine similarity.

This is where most homegrown implementations get messy. You need tokenization, normalization, distance functions, ranking, and some way to handle tie-breaking when scores are identical. You also need to decide whether to lazy-load embeddings or block rendering, how to handle stale indexes, and whether to precompute query embeddings for common searches.

Building It With altor-vec

altor-vec handles the entire client-side pipeline. It ships with a quantized MiniLM model that runs in WASM, a prebuilt indexing CLI, and an API designed for static sites.

Start by installing it:

npm install altor-vec

During your build, generate embeddings from your content:

import { buildIndex } from 'altor-vec/cli';

await buildIndex({
  input: './content/**/*.md',
  output: './public/search-index.bin',
  model: 'minilm-l6-v2',
  chunkSize: 512
});

This produces a binary file with precomputed embeddings and metadata. The chunkSize parameter controls how you split long documents—512 tokens is a reasonable default for article-length content.

At runtime, initialize the search client:

import { VectorSearch } from 'altor-vec';

const search = await VectorSearch.load('/search-index.bin');

const results = await search.query('how do I cancel my subscription', {
  limit: 5,
  threshold: 0.7
});

results.forEach(result => {
  console.log(result.title, result.score);
});

The threshold parameter filters out low-confidence matches. A score of 0.7 is a reasonable floor—anything below that is usually noise.

Handling Updates and Cache Invalidation

Static indexes go stale. If you add a new article, users with a cached index won't see it until you force a refresh. The simplest solution is versioning: append a hash to your index filename (search-index.a3f9b2c.bin) and update the reference in your HTML. When the filename changes, browsers fetch the new file.

For sites that update frequently, consider splitting your index. Separate stable content (old articles, evergreen docs) from recent additions. Ship a base index and a delta. At runtime, merge them. This keeps cache hit rates high while ensuring new content appears immediately.

altor-vec supports index merging:

const baseIndex = await VectorSearch.load('/search-base.bin');
const deltaIndex = await VectorSearch.load('/search-delta.bin');

const search = VectorSearch.merge(baseIndex, deltaIndex);

Delta files are small (a few KB for a dozen new articles), so they download quickly even on slow connections.

When Not to Use Client-Side Vector Search

This approach breaks down when your corpus exceeds roughly 10,000 documents or 10MB of embeddings. At that scale, download times hurt initial page load, and search performance degrades. You also lose access to server-side features like user-specific ranking, A/B testing, and analytics pipelines that need to log queries.

If your search needs to be personalized, federated across multiple data sources, or updated in real time by users, you need a backend. Client-side search works for static, public content where the entire index can be precomputed and shipped as a build artifact.

Comparing to Alternatives

Algolia and Typesense are faster and more featureful, but they cost money and require API keys. Lunr.js and FlexSearch are excellent for keyword search but don't understand semantics—"bug" and "defect" are unrelated tokens. PageFind (from Cloudflare) is the closest equivalent, but it's optimized for static site generation and tightly coupled to their build pipeline.

altor-vec sits in the gap: semantic search, zero backend, framework-agnostic, small enough to ship on every page. If you're building a documentation site, a personal wiki, or an internal tool where the content is static and the corpus is small, it's the right tool.

FAQ

Can I use client-side vector search for e-commerce product search?

Only if your catalog is small (under 5,000 products). For larger catalogs, the index size becomes prohibitive and you lose the ability to filter by stock, price, and user-specific preferences. A hybrid approach—keyword search on the client, vector reranking on the server—works better.

How do I generate embeddings if I don't have Python installed?

altor-vec includes a Node-based embedding API that uses ONNX runtime. You don't need Python or transformers. The CLI handles everything. If you prefer to precompute embeddings separately, the library accepts any JSON file with a { id, vector, metadata } structure.

What happens if the user's query is in a different language than my content?

Multilingual models like paraphrase-multilingual-MiniLM map queries and documents into a shared vector space. If your content is in English and a user searches in Spanish, the model will still find relevant matches. altor-vec supports swapping models via the model parameter.

Does this work offline?

Yes. Once the index and WASM model are cached, search works without a network connection. This makes it ideal for Progressive Web Apps, Electron apps, and internal tools that need to function in low-connectivity environments.
