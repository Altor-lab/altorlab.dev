Vector Search in React — Complete Tutorial

React developers usually hit the same search problem: keyword search is easy to ship, but it fails when the user phrase and the document phrase do not overlap. altor-vec solves the retrieval side by running HNSW vector search locally in WebAssembly. That means you can keep query latency close to the browser, eliminate per-query billing, and still expose a familiar framework component API. This guide focuses on implementation details rather than marketing claims.

Install altor-vec: npm install altor-vec

The example below uses tiny four-dimensional vectors so the code is runnable as-is and easy to understand. In production you would usually replace those manual vectors with embeddings from a model such as Xenova/all-MiniLM-L6-v2 or a build-time embedding job. The retrieval flow stays the same: install, import the WASM package, create an index, optionally add vectors, then query the engine and map result IDs back to metadata.

Step 1: install and understand the runtime boundary

Start with npm install altor-vec. The package exposes a default init() function that loads the WASM module and a WasmSearchEngine class that loads or builds an HNSW index. The important design question in React is not whether vector retrieval is possible. It is where initialization should live so the index is created once, memory is released intentionally, and queries do not trigger unnecessary work on each re-render or navigation.

Step 2: import the library and create the index

The sample builds an index from a flat Float32Array. That matches the real API from the package README: WasmSearchEngine.from_vectors(flat, dims, m, ef_construction, ef_search). The four HNSW parameters here are conservative defaults for a small browser index. If you precompute a production index offline, you can instead serialize it with to_bytes() and load it using new WasmSearchEngine(bytes).

import { useEffect, useMemo, useState } from 'react';
import init, { WasmSearchEngine } from 'altor-vec';

type Doc = { id: string; title: string; vector: [number, number, number, number] };

const docs: Doc[] = [
  { id: 'react-hooks', title: 'React hooks primer', vector: [1, 0, 0, 0] },
  { id: 'vite-build', title: 'Vite build caching', vector: [0, 1, 0, 0] },
  { id: 'browser-rag', title: 'Browser RAG design', vector: [0, 0, 1, 0] },
];

export function VectorSearchDemo() {
  const [engine, setEngine] = useState<WasmSearchEngine | null>(null);
  const [results, setResults] = useState<Array<{ title: string; distance: number }>>([]);

  useEffect(() => {
    let cancelled = false;

    (async () => {
      await init();
      const dim = 4;
      const flat = new Float32Array(docs.flatMap((doc) => doc.vector));
      const localEngine = WasmSearchEngine.from_vectors(flat, dim, 16, 200, 50);

      docs.push({ id: 'semantic-ui', title: 'Semantic UI patterns', vector: [0.92, 0.05, 0.03, 0] });
      localEngine.add_vectors(new Float32Array([0.92, 0.05, 0.03, 0]), dim);

      if (!cancelled) setEngine(localEngine);
    })();

    return () => {
      cancelled = true;
      engine?.free();
    };
  }, []);

  const queryVector = useMemo(() => new Float32Array([0.95, 0.03, 0.02, 0]), []);

  async function runSearch() {
    if (!engine) return;
    const hits: [number, number][] = JSON.parse(engine.search(queryVector, 3));
    setResults(hits.map(([id, distance]) => ({ title: docs[id].title, distance })));
  }

  return (
    <section>
      <button onClick={runSearch} disabled={!engine}>Search local vectors</button>
      <ul>
        {results.map((hit) => (
          <li key={hit.title}>{hit.title} — distance {hit.distance.toFixed(3)}</li>
        ))}
      </ul>
    </section>
  );
}

Step 3: what the code is actually doing

  1. Install: the project adds altor-vec from npm.
  2. Import: the code imports init and WasmSearchEngine.
  3. Create index: manual vectors are flattened into a single Float32Array and passed into from_vectors().
  4. Add vectors: the example appends one more vector via add_vectors() so you can see incremental updates.
  5. Query: it converts a query vector into a Float32Array, calls search(), parses the JSON response, and maps IDs back to the in-memory document array.

That pattern is stable across browser frameworks because altor-vec is model-agnostic. The framework concerns are mostly lifecycle-related: where to hold the engine instance, how to debounce query creation, and whether embeddings run on the main thread or in a worker. If you keep those concerns separate, semantic retrieval feels surprisingly ordinary.

Performance notes specific to this framework

  • Put embedding generation in a worker or React input latency will suffer long before altor-vec becomes the bottleneck.
  • Keep the engine instance in module or hook state so React re-renders do not rebuild the HNSW graph.
  • Memoize mapped hit objects if you render score badges, snippets, and highlighted fields in a large result list.
Published altor-vec baseline: Chrome p95 retrieval on 10K vectors / 384 dimensions is about 0.6ms, index load is about 19ms, the raw WASM binary is 117KB, and the gzipped WASM payload is 54KB. In real apps, embedding generation and rendering usually cost much more than the vector lookup itself.

When to use client-side vs server-side in React

Client-side: React alone is enough when the corpus is public, updates at build time, and the page needs instant local retrieval.

Server-side: Move retrieval to a server when vectors are private, when you need audit logs or tenant isolation, or when the result set is too large to ship to the browser.

A good rule is simple. If the data is already safe to send to every browser and you mostly care about fast semantic ranking, keep it local. If the search layer also needs to enforce business rules, security boundaries, or complex shared state, put retrieval on the server and let React call it as a normal endpoint.

Production checklist

  • Cache the WASM and serialized index aggressively with versioned asset names.
  • Validate vector dimensions before every search to prevent subtle runtime errors.
  • Keep metadata outside the HNSW graph so result rendering stays flexible.
  • Measure cold start, repeated search latency, and memory on at least one mid-range mobile device.
  • Free the engine explicitly if you unload large indexes on navigation.

Conclusion

React does not require a special semantic-search abstraction. It only needs a clean place to initialize the engine and a disciplined boundary between embedding, retrieval, and UI state. altor-vec gives you a small browser-native ANN core, while the framework handles rendering and ergonomics. If you want a developer-friendly starting point with no backend dependency, this is the shortest path: npm install altor-vec, build or load an index, and search locally.
