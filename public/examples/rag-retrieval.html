Build RAG Retrieval with altor-vec

What you'll build: A local retriever that selects the best supporting chunks before you assemble a RAG prompt.

Bundle cost: 54KB · Query time: <1ms · Backend: None required

Install

npm install altor-vec

The Concept

RAG quality often depends more on retrieval than generation. If you select the wrong chunks, even a strong model will answer badly. Local vector search makes it easy to run the retriever close to the user and inspect the exact supporting passages.

Client-side retrieval is especially appealing for docs assistants, onboarding copilots, and demos where the corpus is static. You can ship the chunk index once and keep prompt assembly deterministic and fast.

The code below is a real retrieval stage that indexes prompt-ready chunks. It stops at retrieval, which is the right boundary if you want to pair local search with either a local generator or a remote LLM fallback.

This page uses a deterministic embedding helper so the snippets are runnable with only altor-vec. For production ranking quality, replace that helper with a real embedding model or precomputed vectors.

Full Code Example

import init, { WasmSearchEngine } from 'altor-vec';

const dims = 8;
const records = [
  {
    "title": "HNSW search parameter",
    "text": "ef_search increases candidate breadth at query time and usually improves recall.",
    "meta": "chunk-1"
  },
  {
    "title": "HNSW construction parameter",
    "text": "ef_construction affects build quality and build-time cost.",
    "meta": "chunk-2"
  },
  {
    "title": "Memory tradeoff",
    "text": "Higher M improves graph connectivity but increases memory usage.",
    "meta": "chunk-3"
  },
  {
    "title": "Embedding normalization",
    "text": "Normalized vectors make dot product equivalent to cosine similarity.",
    "meta": "chunk-4"
  },
  {
    "title": "Browser caching",
    "text": "Versioned index assets let local retrieval work offline and stay cache-safe.",
    "meta": "chunk-5"
  },
  {
    "title": "Hybrid fallback",
    "text": "Use local retrieval for hot data and server retrieval for large long-tail corpora.",
    "meta": "chunk-6"
  }
];

function embedText(text) {
  const vector = new Float32Array(dims);
  for (const token of text.toLowerCase().split(/[^a-z0-9]+/).filter(Boolean)) {
    let hash = 2166136261;
    for (const char of token) hash = Math.imul(hash ^ char.charCodeAt(0), 16777619);
    const slot = Math.abs(hash) % dims;
    vector[slot] += 1;
    vector[(slot + token.length) % dims] += token.length / 10;
  }
  const magnitude = Math.hypot(...vector) || 1;
  return Array.from(vector, (value) => value / magnitude);
}

async function main() {
  await init();

  const flat = new Float32Array(
    records.flatMap((record) => embedText(`${record.title} ${record.text} ${record.meta}`))
  );

  const engine = WasmSearchEngine.from_vectors(flat, dims, 16, 200, 64);
  const hits = JSON.parse(engine.search(new Float32Array(embedText('how does ef_search affect recall')), 4));

  const results = hits.map(([id, distance]) => ({
    ...records[id],
    similarity: Number((1 - distance).toFixed(3)),
  }));

  console.table(results);
  engine.free();
}

main();

  const vector = new Float32Array(dims);
  for (const token of text.toLowerCase().split(/[^a-z0-9]+/).filter(Boolean)) {
    let hash = 2166136261;
    for (const char of token) hash = Math.imul(hash ^ char.charCodeAt(0), 16777619);
    const slot = Math.abs(hash) % dims;
    vector[slot] += 1;
    vector[(slot + token.length) % dims] += token.length / 10;

  const magnitude = Math.hypot(...vector) || 1;
  return Array.from(vector, (value) => value / magnitude);

export function RAGRetrievalExample() {
  const [engine, setEngine] = useState(null);
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  useEffect(() => {
    let cancelled = false;
    let instance;

    (async () => {
      await init();
      const flat = new Float32Array(
        records.flatMap((record) => embedText(`${record.title} ${record.text} ${record.meta}`))
      );
      instance = WasmSearchEngine.from_vectors(flat, dims, 16, 200, 64);
      if (!cancelled) setEngine(instance);
    })();

    return () => {
      cancelled = true;
      instance?.free();
    };
  }, []);

  useEffect(() => {
    if (!engine || query.trim().length < 2) {
      setResults([]);
      return;
    }

    const hits = JSON.parse(engine.search(new Float32Array(embedText(query)), 5));
    setResults(
      hits.map(([id, distance]) => ({
        ...records[id],
        similarity: Number((1 - distance).toFixed(3)),
      }))
    );
  }, [engine, query]);

  return (
    <section>
      <input
        value={query}
        onChange={(event) => setQuery(event.target.value)}
        placeholder="Ask the retriever a question"
      />
      <ul>
        {results.map((result) => (
          <li key={result.title}>
            <strong>{result.title}</strong> — {result.meta} (score {result.similarity})
          </li>
        ))}
      </ul>
    </section>
  );
}

How It Works

The flow is the same pattern you would use in production: turn each record into an embedding vector, build an HNSW graph with WasmSearchEngine.from_vectors(), and query it with the embedding for the current search text. HNSW keeps approximate nearest-neighbor search fast by traversing a layered graph instead of scanning every vector linearly.

Similarity scoring here is derived from the returned distance with 1 - distance so the UI can show an intuitive confidence number. If you swap in a real embedding model such as MiniLM, multilingual-e5, or precomputed server-generated vectors, the surrounding indexing and querying code stays almost identical.

When to Use This Pattern

  • Browser-based docs assistants
  • Product onboarding copilots
  • Local knowledge tools bundled into desktop apps
  • Privacy-sensitive QA systems where retrieval should stay on-device

Limitations

RAG still needs a separate embedding pipeline and often a generator. Larger corpora or access-controlled content may require hybrid architectures, and evaluation should include retrieval precision, not just chat transcripts.
